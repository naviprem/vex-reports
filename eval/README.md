# eval/ — the learning-loop regression suite & run records

Companion to `docs/pilot/learning-loop-protocol.md`. This is the **Grader/Improver**
side of the loop; it holds the answers and must never be visible to a Runner session
(protocol §2).

## Layout

| Path | Role |
|---|---|
| `suite/<CANONICAL-ID>__<component>__<image>.md` | One expected-verdict entry per finding per component copy (protocol §1 format), including a `confidence` field (`high`/`medium`/`low`) recording how far the ground truth is independent of the adjudicator — it gates the −10 (protocol §4). Seeded from pilots 1–2 (13 findings); grows every pilot. |
| `runs/<pilot-slug>/` | A Runner's `vex-out/` copied back here for grading. One dir per pilot run. |
| `learnings-log.md` | Append-only: one entry per miss (symptom → root cause → rule → diff → regression result → accepted/reverted). |
| `target-matrix.md` | The §6 coverage matrix; each pilot's cell assignments + Phase A/C candidate picks. |

## The physical separation (why runs/ exists)

The Runner works in an isolated workspace **outside this repo** (default
`~/triage-runs/<slug>/`), scaffolded by `scripts/freeze-skill.sh`. It sees only a
frozen copy of the skill and the input bundle — never this directory. After it
writes `vex-out/`, copy that into `eval/runs/<slug>/` and grade it here against
`suite/`.

```
scripts/freeze-skill.sh pilot-03-django-debian   # freeze skill -> ~/triage-runs/
# ... drop triage-input/, run a fresh `claude` session there, it writes vex-out/ ...
cp -R ~/triage-runs/pilot-03-django-debian/vex-out eval/runs/pilot-03-django-debian
# ... grade eval/runs/pilot-03-.../ against eval/suite/ ...
```

## Running a pilot — the blind run, step by step

The steps below assume the adjudicator has already **adjudicated-first** (written the
expected `suite/` entries) and run `scripts/freeze-skill.sh <slug>`, which creates
`~/triage-runs/<slug>/` with `skill/`, `triage-input/`, and an empty `vex-out/`. The
adjudicator then populates `triage-input/` with the input bundle (findings export +
app repo at the build commit + image ref).

1. **Open a fresh Runner session** — a *new* terminal, never the adjudicator's session
   (that one knows the answers). A fresh session in the workspace has no filesystem path
   to `eval/`, so it is structurally blind:
   ```
   cd ~/triage-runs/<slug> && claude
   ```

2. **Give it the kickoff prompt.** The skill there is plain files, not an installed
   `/vex-draft` command, so the Runner runs it by reading and following it:
   ```
   Read RUNNER-README.md, then skill/skills/vex-draft/SKILL.md and its
   skill/references/. Execute that vex-draft triage on the bundle in
   triage-input/ (findings.json + repo/ at the build commit). Walk the
   evidence ladder, and write all evidence files + CSAF drafts + the
   summary verdict table to vex-out/. You have no access to expected
   answers — do not look for them.
   ```

3. **Let it run.** It walks the ladder blind and writes `vex-out/`.

4. **Copy the output back to grade:**
   ```
   cp -R ~/triage-runs/<slug>/vex-out eval/runs/<slug>
   ```

5. **Grade** `eval/runs/<slug>/` against `suite/` using the §4 scoring in
   `../docs/pilot/learning-loop-protocol.md`, then the improver proposes one edit per
   failure class. Record misses in `learnings-log.md`.

> The Runner's own copy of steps 2–3 (its job + the do-nots) lives in
> `~/triage-runs/<slug>/RUNNER-README.md`, generated by the freeze script.
